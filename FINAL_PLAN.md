세부 영역별 배분은 `TOPIC_MATRIX.md` 참조 (1과목 11개 영역, 2과목 19개 영역).

### 2.2 데이터 스키마 (확정 v2)

```javascript
// questions.js 예시
const questions = [
  {
    id: 1,                    // 전역 ID (1~500)
    subj: 1,                  // 과목 (1 또는 2)
    topic: "1-F",             // 토픽 영역 코드 (TOPIC_MATRIX.md 기준)
    topic_name: "정규화",      // 영역 이름 (필터링/검색용)
    diff: "상",               // 난이도 (하/중/상)
    has_code: false,          // 코드 문제 여부 (UI 분기용)
    
    q: "본문 텍스트만. 표/코드는 blocks 배열로.",
    
    blocks: null,             // 표/코드 블록 배열 (선택, null 가능)
    // blocks 예시:
    // [
    //   { type: "table", title: "[EMP]", headers: [...], rows: [[...]] },
    //   { type: "code",  title: "[쿼리]", lang: "sql", content: "SELECT ..." },
    //   { type: "table", title: "[결과]", headers: [...], rows: [[...]] }
    // ]
    // 블록 타입: "table" | "code"
    // null 값은 사이트에서 "NULL"로 표시
    
    choices: ["1번", "2번", "3번", "4번"],
    ans: 3,                   // 정답 번호 (1~4)
    src: "자료3 p.13~14",      // 자료 출처
    exp: {                    // 4단 해설
      reason: "정답 + 한 줄 근거",
      terms: ["**용어1**: 설명", "**용어2**: 설명"],
      wrong: ["1. 1번이 틀린 이유", "3. ...", "4. ..."],
      tip: "한 줄 암기 팁"
    }
  }
];
```

**blocks 배열 사용 가이드**:
- 표 없는 문제 (대부분 개념 문제): `blocks: null`
- 표 1개 + SQL: `[{type:"table"}, {type:"code"}]`
- 표 + SQL + 결과 표 (빈칸 채우기 등): `[{type:"table"}, {type:"code"}, {type:"table"}]`
- 결과 표만 + SQL 빈칸: `[{type:"table"}, {type:"code"}]` (결과를 먼저 보여주는 형태)
- SQL 코드 두 개 비교: `[{type:"code", title:"SQL A"}, {type:"code", title:"SQL B"}]`

블록 순서는 문제별로 자유. 사이트는 배열 순서대로 렌더링.

### 2.3 해설 구조 (확정 v3)
4단 간소화 구조:
1. **정답 + 한 줄 근거** (자료 페이지 명시)
2. **용어 / 함수 동작** (해당 문항에 등장한 핵심 용어 1~5개)
3. **오답 분석** (선지별 1줄)
4. **암기 팁** (한 줄 압축)

코드 문제는 용어 단에 함수 시그니처/처리 순서 포함.

### 2.4 보기 구성 5원칙 (확정)
1. 선지의 차이는 "한 함정의 적용 여부"여야 한다
2. 오답에는 흔한 오개념을 박는다
3. 인접 개념 보기를 섞는다 (예: 1NF/2NF/3NF/BCNF 모두 등장)
4. 숫자만 살짝 다른 결과 (160/400, 'Dev'/'eve' 등)
5. 정답을 모르는 사람이 합리적으로 추론해도 50% 이상 헷갈려야 한다

## 3. 사이트 구현 계획

### 3.1 기술 스택 (빅분기CBT 포팅)
- **단일 HTML 파일** (`index.html`) — 빌드 도구 없음, 의존성 없음
- **순수 JavaScript** (프레임워크 없음)
- **CSS 변수 기반 다크/라이트 테마**
- **localStorage**로 진행 상태/오답 노트 저장
- **별도 데이터 파일** `questions.js` — 500문항 적재

### 3.2 화면 구성 (4화면)

| 화면 | 역할 |
|------|------|
| `home` | 시작 화면, 학습 모드 선택, 통계 |
| `quiz` | 문제 풀이 (타이머, 50문항) |
| `result` | 결과 (점수, 합격/불합격, 과목별 점수) |
| `review` | 오답 풀이 (해설 표시) |

### 3.3 핵심 기능

#### 3.3.1 모드 (사용자 결정 필요)
- **모의고사 모드**: 50문항 무작위(1과목 10 + 2과목 40), 90분 타이머
- **영역별 학습 모드**: 토픽 영역 선택해서 그 영역만 풀기 (예: "정규화만 20문항")
- **오답 노트 모드**: 이전에 틀린 문항만 다시 풀기
- **랜덤 학습 모드**: 영역 무관 N문항 랜덤

#### 3.3.2 코드 문제 렌더링
`has_code: true` 문항은 SQL 코드와 데이터 테이블을 정렬된 모노스페이스로 표시.
```html
<pre class="code-block"><code>SELECT COUNT(*) FROM EMP ...</code></pre>
<table class="data-table">...</table>
```
구문 강조는 단순 정규식 기반 (CSS 클래스로 처리).

#### 3.3.3 해설 렌더링
4단 구조를 카드 형식으로 표시. 각 단을 접기/펴기 가능하게 하면 학습 효율 ↑.

#### 3.3.4 진행 상태 저장
`localStorage`에 다음 저장:
- 풀이 중인 문제 번호와 답안
- 누적 통계 (전체 정답률, 영역별 정답률)
- 오답 노트 (틀린 문항 ID 리스트)

### 3.4 스키마 차이점 (빅분기CBT 대비)
| 필드 | 빅분기CBT | SQLD |

### 4.2 양산 분담 (모드 C 에이전트팀)

**팀 구성**: 5~6명 + 검증 에이전트 1명

| 에이전트 | 담당 영역 | 문항 수 |
|---------|----------|---------|
| `gen-1gwa` | 1과목 전체 (모델링/식별자/정규화 등) | 100 |
| `gen-2-select` | 2과목 SELECT/함수/WHERE | 약 75 |
| `gen-2-join` | 2과목 조인/서브쿼리 | 95 |
| `gen-2-window` | 2과목 그룹/윈도우/Top N/계층형 | 약 110 |
| `gen-2-pivot-dml` | 2과목 PIVOT/정규식/DML/TCL | 약 50 |
| `gen-2-ddl` | 2과목 DDL/뷰/제약/DCL | 약 70 |
| `validator` | 전체 검증 (정답 정확성, 형식 통일, PostgreSQL 실행) | - |

**병렬 양산**: 6명이 동시에 작업하면 실질적으로 **2~4시간 내 1차 초안 완성**.

### 4.3 검증 정책 (정확성 보장)

**4.3.1 코드 문제 (약 280문항)**
- 모든 SQL을 PostgreSQL `sqld_verify` DB에서 실행
- 결과를 보기 4번 중 정답 보기와 정확히 일치하는지 확인
- Oracle 전용 함수(NVL/DECODE/ROWNUM/CONNECT BY 등)는 자료 인용 기반 + 가능한 한 표준 SQL 동등 변환으로 교차 확인
- 검증 결과는 `validator`가 별도 로그 파일에 기록

**4.3.2 개념 문제 (약 220문항)**
- 모든 정답에 자료 페이지 명시(`src` 필드)
- 검증 에이전트가 해당 페이지를 직접 읽고 정답 일치 여부 확인
- 자료에 명시되지 않은 추론 정답 금지 (사용자 강조 사항)

**4.3.3 검증 실패 시**
- 정답이 자료와 불일치 → 문항 폐기 또는 정답 수정
- 코드 결과 불일치 → 보기 재구성 또는 시나리오 변경

### 4.4 형식 통일 체크리스트

- [ ] 보기 4개 모두 그럴듯한 함정
